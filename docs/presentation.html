🏦 Mastercard Cloud Edge + AWS Payment Cryptography

发卡方（Issuer）POC 演示

Mastercard Cloud Edge AWS Payment Cryptography ap-southeast-1

📋 演示背景

Mastercard 于 2025 年 6 月宣布推出 Cloud Edge 平台，与 AWS 合作让亚太地区银行和金融科技公司通过云端快速接入 Mastercard 支付网络，onboarding 时间缩短 4 倍。

本 POC 演示发卡方如何利用 AWS Payment Cryptography 替代传统物理 HSM，完成支付交易的密码学验证。

价值对比

传统方式Cloud Edge + AWS 方式
采购物理 HSM，数月交付API 调用即用，分钟级开通
物理机房 + 专线接入AWS PrivateLink 私有连接
HSM 固件升级需人工操作全托管，AWS 负责维护

🎬 演示场景一：交易授权（实时）

展示一笔交易从发起到授权的完整密码学处理链路。

电商无卡交易 (Card Not Present)

持卡人在线购物，商户将卡号+有效期+CVV2 发送到 Processor 验证。

Step 1：CVV2 验证通过 → 授权

持卡人在 HKTVmall 网购 HKD 299，输入正确 CVV2。

POST /authorize
{
  "transaction_type": "ecommerce",
  "pan": "5425230000004415",
  "amount": 299, "currency": "HKD",
  "merchant": "HKTVmall",
  "expiry_date": "0127", "cvv2": "683"
}

预期结果：✅ 00 - Approved

Step 2：CVV2 错误 → 拒绝（欺诈拦截）

盗卡者猜测 CVV2 发起交易，Processor 即时拦截。

POST /authorize
{
  "transaction_type": "ecommerce",
  "pan": "5425230000004415",
  "amount": 1200, "currency": "HKD",
  "merchant": "Suspicious Store",
  "expiry_date": "0127", "cvv2": "999"  ← 错误
}

预期结果：❌ 05 - Do not honour

3D Secure（CAVV）— 在线交易额外认证

🔲 支持，未演示 — 电商交易中验证持卡人身份的额外密码学步骤。后续版本补充。

非接触式支付 (Contactless / Tap-to-Pay)

芯片卡每笔交易动态生成 dCVV2，基于 ATC（交易计数器）派生，防止重放攻击。

Step 1：dCVV2 动态验证通过

持卡人 Tap-to-Pay 在港铁充值 HKD 88，芯片生成当次交易的 dCVV2。

POST /authorize
{
  "transaction_type": "contactless",
  "pan": "5425230000004415",
  "amount": 88, "currency": "HKD",
  "merchant": "Octopus Top-up MTR",
  "expiry_date": "0127", "pan_sequence": "00",
  "atc": "0001",  ← 当前交易计数器
  "service_code": "101", "dcvv2": "166"
}

预期结果：✅ 00 - Approved

Step 2：重放攻击拦截 — ATC 不匹配

攻击者截获历史交易报文重放，但 ATC 已递增，验证失败。

POST /authorize
{
  "transaction_type": "contactless",
  ...
  "atc": "0009",  ← 篡改的 ATC（实际应为 0002）
  "dcvv2": "166"  ← 旧的 dCVV2
}

预期结果：❌ 05 - Do not honour (replay attack)

POS 刷卡 — 完整 PIN 处理链路

一笔 POS 交易从收单方到发卡方的完整密码学处理：MAC 验证 → PIN 翻译 → PIN 验证。这是 Card Processor 最核心的操作。

完整链路：MAC 验证 → PIN 翻译 → PIN 验证

持卡人在百佳超市 POS 刷卡消费 HKD 2,000，输入 PIN。

POST /authorize
{
  "transaction_type": "pin_translate",
  "pan": "5425230000004415",
  "amount": 2000, "currency": "HKD",
  "merchant": "POS Terminal - Park N Shop",
  "encrypted_pin_block": "3E15A2191F11D647",  ← 收单方密钥加密
  "pin_verification_value": "7914",
  "message_data": "0200542523000044150000000050001234567890",
  "mac": "C8288190"  ← 报文 MAC
}

处理流程：
① MAC 验证 → 确认报文未被篡改
② PIN Translate → 收单方密钥转换为发卡方密钥
③ PIN Verify → 用发卡方密钥验证 PIN 正确性

预期结果：✅ 00 - Approved (3步全部通过)

MAC 篡改检测 — 报文被中间人修改

攻击者篡改了交易金额（$50→$9999），MAC 校验失败，交易在第一步即被拦截。

POST /authorize
{
  "transaction_type": "mac_verify",
  "pan": "5425230000004415",
  "amount": 9999,
  "message_data": "0200542523000044150000000099991234567890",  ← 金额被篡改
  "mac": "C8288190"  ← 原始 MAC（与篡改后的消息不匹配）
}

预期结果：❌ 96 - MAC verification failed

🎬 演示场景三：运营支撑（低频）

日常运营中的敏感数据保护、密钥交换与 API 安全。

敏感数据保护与密钥交换

Processor 日常运营中需要加密存储敏感数据（如 PAN），以及与卡组织/收单方安全交换密钥。

PAN 加密 — 敏感数据落盘保护

将持卡人卡号加密后存储，明文 PAN 不落盘，满足 PCI DSS 要求。

POST /authorize
{
  "transaction_type": "encrypt",
  "pan": "5425230000004415"
}

→ 返回密文，可安全存储到数据库

预期结果：✅ 返回 CipherText

TR-31 密钥导出 — 与外部系统交换密钥

将 MAC 密钥以 TR-31 Key Block 格式导出，用于与卡组织或收单方安全交换。

POST /authorize
{
  "transaction_type": "key_export",
  "key_alias": "poc-mac-key"
}

→ 返回 TR-31 Key Block（可安全传输给对方导入）

预期结果：✅ 返回 Key Block + KCV

API 安全防护

无 API Key — 未授权请求被拒

模拟未授权的调用方，不携带 API Key 直接请求。

# 无 x-api-key header → 403
POST /authorize  (无认证)

预期结果：🚫 403 - Forbidden
